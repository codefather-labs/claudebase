## Conventions

- **`Verification Class`** — one of `UI/UX`, `API`, `DB`, `CLI`, `FS`, `Mixed`. KP1-KP3 are `Mixed` per plan §"Success Criteria" evidence table (`.claude/plan.md` lines 52-56). Bot-command cases are `Mixed` (UI/UX + DB). Token / Dispatcher error cases are `Mixed` (CLI + FS). Edge cases are `Mixed` (DB + FS + UI/UX).
- **`Evidence Required`** — concrete artifact names. Vague entries ("works correctly", "no errors") are auto-FAIL at qa-engineer execution time per `~/.claude/rules/cognitive-self-check.md`.
- **Screenshot mechanism** — Playwright MCP cannot capture native Telegram Desktop (plan §"Success Criteria" Plan-Critic finding #6, plan line 58). Live-run evidence uses OS-level screenshots: PowerShell `Get-Screenshot` / Snipping Tool on Windows, `screencapture` on macOS, `gnome-screenshot` on Linux. Telegram Web in a Playwright-driven browser is an acceptable alternate path.
- **Evidence storage** — screenshots, logs, and SQL outputs live under `docs/qa/evidence/<tc-id>-*` (convention). The qa-engineer creates the directory at first run.
- **Routing-key placeholders** in evidence: `<dm_chat_id>` = operator's DM chat_id with bot `@X`; `<G>` = group chat_id; `<α_id>` = forum topic α id; `<β_id>` = forum topic β id; `<γ_id>` = unbound forum topic γ id (TC-11). The qa-engineer substitutes concrete integers at runtime.
- **Log-tail command** — `claudebase daemon logs --tail 50` returns the last 50 lines of the daemon's log stream (Linux/macOS via journalctl-equivalent; Windows via Event Log or in-process tail). The literal substrings cited in `Expected Result` and `Evidence Required` columns MUST appear verbatim.
- **SQL command** — `sqlite3 ~/.claude/knowledge/chat.db "<query>"`. Path resolved per the §17 `OQ-ACD-4` resolution (user-level `chat.db`).
- **Bot token** — KP1-KP3 / TC-4 through TC-7 / TC-11 through TC-15 require `TELEGRAM_BOT_TOKEN` set and bot `@X` added to group `G` with Forum Topics permission. TC-8 / TC-9 / TC-10 are deliberate-failure paths.
